# DevOps Server 实施 High Availability（增加 AT）

备注：仅新增 Application Tier（AT）时，比如从 1 台 AT => 2 台 AT（接入负载均衡），通常可以做到“近零停机”。

适用范围：
- 已有单 AT 的 Azure DevOps Server（版本不变）
- 目标是在不迁移数据库的前提下，扩容 AT 并接入 LB

不适用范围：
- 同时进行硬件迁移、跨域迁移、版本升级（建议拆分变更窗口）

变更目标：
- 提升可用性（AT 单点消除）
- 保持现网 URL 不变（通过 DNS/LB 切流）
- 控制业务中断在分钟级（主要发生在切流瞬间）

---

## 1. 变更前检查清单（必须逐项确认）

### 1.1 版本与拓扑

- [ ] 新 AT 操作系统与补丁级别满足 Azure DevOps Server 要求
- [ ] 新 AT 安装包版本、Patch 与现网一致
- [ ] 现网 AT、SQL、Search、Build Agent 运行正常，无积压告警

### 1.2 账号与权限

- [ ] 安装账号在新 AT 上具有本地管理员权限
- [ ] Azure DevOps 服务账号密码可用（如域账号，确认未过期）
- [ ] LB/DNS 操作账号具备变更权限

### 1.3 网络与证书

- [ ] 新 AT 到 SQL/AD/SMTP/代理等依赖端口可达
- [ ] 新 AT 已导入与现网一致的 HTTPS 证书（如启用 TLS）
- [ ] 防火墙已放通 LB 到新 AT 的业务端口与健康检查端口

---

## 2. 准备新的 AT

- 在新 AT 上安装同版本的 Azure DevOps Server
- 在安装配置时，选择加入已有 Deployment（不要新建 Deployment）
- 按现网标准完成以下配置对齐：
    - 服务账号
    - 证书绑定
    - 代理/防火墙白名单
    - 必要扩展或任务组件（如有）

此时，不影响现网环境。我们可以在这台新 AT 上验证（localhost）

建议执行：
- 打开新 AT 的本机门户并完成登录验证
- 检查 Job 是否正常拉起（无持续失败）
- 查看应用日志是否存在认证、数据库连接、证书相关错误

---

## 3. LB 接入前验证（建议）

- 使用 hosts 临时将测试终端指向新 AT，验证关键路径：
    - 仓库读取与提交
    - Work Item 查询与更新
    - Pipeline 队列与日志读取
- 检查 `_apis/health` 返回状态正常
- 压测建议：小流量并发登录/克隆，验证响应无异常抖动

## 4. 接入负载均衡

- 推荐 Azure LB
    - 配置健康检查（`/_apis/health`）
    - 配置 Sticky Session（推荐）
    - 说明：如不启用，会话可能在 AT 切换时重建，用户可能出现重新登录；长连接（Git/Agent）体验会受影响
- 将所有的 AT 都加入到 LB

LB 配置建议：
- 健康检查间隔：5-15 秒
- 连续失败阈值：2-3 次
- 新节点先灰度（仅接入少量流量）再全量

---

## 5. 切流执行步骤（操作手册）

### 5.1 切流前 30 分钟

- [ ] 通知业务冻结高风险操作（大规模导入、批量迁移、关键发布）
- [ ] 将 DNS TTL 临时降到 60-300 秒（若当前 TTL 较大）
- [ ] 记录基线：当前登录成功率、API 错误率、Agent 在线数

### 5.2 执行切流

1. 更新 DNS：Client -> LB VIP
2. 观察 5-10 分钟：
     - 健康探测成功率
     - 4xx/5xx 是否异常上升
     - 用户登录与 Git 操作是否正常
3. 若无异常，提升新 AT 权重至目标值（或加入全量后端池）

## 5.3 切流说明

- DNS 解析切换（原来的 Client -> AT/IP，变成 Client -> LB/IP -> AT）
- 切换过程可能出现短时网络波动（连接重置/短时解析不一致），通常为分钟级

---

## 6. 切流后验收清单

### 6.1 功能验收

- [ ] 用户可正常登录（含 SSO 场景）
- [ ] Git Clone/Pull/Push 正常
- [ ] Work Item 新建/编辑/查询正常
- [ ] Pipeline 可排队、可执行、日志可查看

### 6.2 稳定性验收

- [ ] 新 AT CPU/内存/磁盘无异常尖峰
- [ ] 应用日志无持续错误（认证、数据库连接、Job）
- [ ] LB 健康检查稳定，后端无频繁摘除

### 6.3 可用性验收（建议）

- [ ] 人工下线任一 AT，确认业务可持续
- [ ] 恢复下线 AT 后，健康检查自动恢复

---

## 7. 回退方案（分钟级）

触发条件（任一满足）：
- 连续 10 分钟关键业务失败率明显升高
- 大量用户无法登录或无法提交代码
- 新 AT 持续健康检查失败且无法快速修复

回退步骤：
1. DNS 回切到原 AT（使用切流前记录值）
2. LB 后端池移除新 AT 或将其权重降为 0
3. 验证核心业务恢复（登录、Git、Work Item、Pipeline）
4. 记录故障时间线与日志，进入问题复盘

说明：本方案仅扩容 AT，不涉及数据库结构变更，因此通常无需 DB 回滚。

---

## 8. 变更窗口示例

| 时间点 | 动作 | 预计耗时 |
| --- | --- | --- |
| T-30m | 预检查、通知、TTL 调整 | 30m |
| T-0 | DNS 切流到 LB | 5m |
| T+5m | 功能与监控验证 | 15m |
| T+20m | 新 AT 全量接入 | 10m |
| T+30m | 稳定性观察与宣告完成 | 30m |

---

## 9. 常见问题与建议

1. 不建议一次窗口同时做“扩容 + 升级 + 跨域”三件事。
2. 若 LB 健康检查路径错误，可能导致节点被全部摘除。
3. 若证书链不完整，新 AT 常出现登录跳转或 TLS 报错。
4. 切流后前 30 分钟建议重点观察 Agent 与 Git 长连接行为。
